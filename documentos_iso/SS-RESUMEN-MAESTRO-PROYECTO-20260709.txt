============================================================
1. IDEA CENTRAL
============================================================

Save Swimmer es un wearable acuatico de seguridad para nadadores de aguas abiertas.

No nace como reloj deportivo ni como monitor fitness. La finalidad principal es seguridad:
detectar comportamiento corporal compatible o incompatible con natacion funcional, registrar
datos y, a futuro, emitir alertas a entrenador, familia o sistema de rescate segun el contexto.

Premisa permanente:
La prioridad es encontrar y asistir al nadador. Las metricas deportivas existen para dar valor,
uso frecuente e interes al producto, pero no deben desplazar la finalidad de seguridad.

Ubicacion fisica prevista:
Espalda alta / entre escapulas.

Motivo:
- Mejor estabilidad biomecanica.
- Mejor lectura global del torso.
- Menor ruido que muneca o cabeza.
- Mejor posicion para GPS si el nadador esta cerca de superficie.
- Posible visibilidad LED posterior/lateral.

2. VERSIONES CONCEPTUALES
============================================================

SAVE SWIMMER LITE
- Enfoque local y masivo.
- Para entrenamientos, nado recreativo y aguas abiertas locales.
- BLE para configuracion y descarga cercana.
- GPS integrado.
- Registro local en microSD durante desarrollo.
- Posible GSM/LTE en version comercial o version Lite Connect.
- Membresia estimada inicial pensada: S/ 40 a S/ 50 mensuales.
- Valor de referencia deseado del dispositivo: S/ 600 a S/ 800, no mandatario.

SAVE SWIMMER PRO
- Mayor cobertura.
- Posible conectividad satelital.
- Mayor autonomia o bateria.
- Dispositivo mas grande.
- Membresia mayor por costos de cobertura.

SAVE SWIMMER EVENT
- Para competencias y eventos.
- Puede usar RF/LoRa u otra infraestructura local.
- No debe mezclarse todavia con la comunicacion comercial de la version base.
- En eventos, el primer contacto de emergencia es el organizador/coach/rescate del evento.
- Posible modelo futuro: registro QR, asignacion de dispositivo, geocercas por evento, historial.

APP FAMILIA
- Debe ser simple.
- Probablemente gratuita.
- Recibe ubicacion/estado/alerta cuando el atleta comparte o inicia actividad.
- No debe saturarse con telemetria tecnica.

APP COACH
- App monitor.
- No debe apropiarse del historial completo del atleta salvo autorizacion.
- En entrenamiento grupal, el coach es primer contacto ante emergencia.
- Debe poder ver multiples atletas activos, mapa, estado, alertas y geocerca de sesion.
- SOS y alertas de seguridad siempre deben existir aun en version gratuita.

APP ATHLETE
- Historial del atleta.
- Lectura de sesion.
- Resumen de rotacion, alineacion, impulso, ritmo, brazadas, GPS y evolucion.
- Permite compartir datos anteriores con coach o familia si el atleta lo decide.

5. CRITERIOS DE LECTURA CORPORAL ACTUALES
============================================================

Datos crudos:
- LR
- FB
- UD
- MAG
- GPS
- Energia INA219

Interpretacion en desarrollo:

ROTACION
- Ya no debe verse como simple LR aislado.
- La lectura mas estable actual usa combinacion FB/UD contra base:
  rotacion aproximada = atan2(FB, UD) ajustada por base.
- La rotacion buscada para nado libre / aguas abiertas se tomo como referencia:
  32 a 45 grados por lado.
- Mas de 50 grados puede interpretarse como sobre-rotacion.
- Menos de 32 grados puede sugerir poca rotacion del torso.

ALINEACION
- Se interpreta como linea corporal / cabeza-cuerpo.
- La lectura actual usa combinacion -LR/UD contra base:
  alineacion aproximada = atan2(-LR, UD) ajustada por base.
- No se debe interpretar literalmente como cabeza porque el sensor esta en espalda alta,
  pero puede dar pistas de perdida de linea corporal, elevacion indebida o esfuerzo.

IMPULSO / EMPUJE
- MAG sirve como energia de movimiento.
- No equivale automaticamente a avance.
- Puede marcar impulso, cambio brusco, freno o perdida de suavidad.
- Se debe comparar con GPS/velocidad cuando exista.
- Una brazada puede tener menos pico de MAG si el nadador ya esta en ritmo constante.

RITMO / BRAZADAS
- La app puede detectar ciclos aproximados por patrones de rotacion.
- Se debe contemplar:
  - Brazada rapida pero poco eficiente.
  - Brazada lenta pero estable.
  - Diferencias entre piscina, aguas abiertas, velocidad y fondo.

RESPIRACION
- Todavia no esta confirmada como metrica directa.
- Podria inferirse si hay patrones repetidos de alineacion/rotacion asociados a toma de aire.
- No debe prometerse hasta tener datos de agua.

SIMETRIA
- Se busca comparar lado derecho e izquierdo.
- No solo media general: deben analizarse picos por lado.
- El usuario noto que si se promedia contra 0 o negativo, el resultado puede ocultar picos reales.
- Pendiente afinar deteccion de picos por lado y graficos.

BASE / CALIBRACION
- Mesa plana sirve como base de hardware.
- Base de colocacion sirve para compensar posicion real en el atleta.
- El prototipo debe aprender con datos:
  - mesa / reposo
  - parado
  - nadando
  - transiciones
- A futuro no debe exigir que el atleta flote estatico para calibrar.

7. RESULTADOS DE COMUNIDAD / FEEDBACK
============================================================

Feedback recibido:
- Medir ritmo de brazadas y velocidad.
- Ubicacion en tiempo real.
- Salida de zona segura / geocerca.
- Deteccion de inmovilidad.
- Patron anormal de movimiento.
- Boton SOS rapido.
- Orientacion/ruta visible porque nadadores se desorientan.
- Confirmacion de salida del agua.
- Distancia a costa/base.
- Alertar entrenador/familia.
- Preocupacion por comodidad y precio.
- Interes en analisis de atleta.

Diferencial observado al conversar:
- Cuando se explica "alerta temprana automatica" la gente presta mas atencion.
- Relojes existentes pueden compartir trayectoria, pero muchos dependen de accion manual.
- Save Swimmer debe diferenciarse por contexto acuatico + alerta automatica + seguimiento.

10. CONECTIVIDAD FUTURA
============================================================

BLE:
- Configuracion cercana.
- Descarga diagnostica.
- No ideal para descarga larga.

GSM/LTE:
- Necesario para version con ubicacion en vivo sin depender de telefono.
- Se debe evaluar:
  - SIM nacional a nombre de Save Swimmer.
  - SIM internacional para usuarios que viajan.
  - Cobertura real en mar, a nivel 0 msnm.
  - Consumo.
  - Tiempo desde recuperacion de senal hasta envio.
  - Buffer local cuando no hay senal.

LoRa/RF:
- Eliminado de version Lite publica por falta de receptor en playa.
- Puede volver en Event.

Satelital:
- Para Pro, no para Lite inicial.

Gateway con telefono:
- Probado como simulacion casi final:
  ESP por BLE -> telefono A -> datos moviles -> backend/telefono B.
- Sirve para pruebas de campo y coach.
- Problemas:
  - Si pantalla se apaga puede cortar BLE.
  - Datos pueden tener delay.
  - ngrok puede limitar cuota.

12. PENDIENTES TECNICOS IMPORTANTES
============================================================

Hardware:
- Sensor de agua.
- Encendido magnetico/reed + circuito de corte real.
- Mejor gestion energia.
- Bateria adecuada final.
- LED 360 o sistema visible.
- Buzzer/vibracion solo si aporta y no complica waterproofing.
- Carcasa o maqueta sellada para piscina.
- Antena GPS despejada.
- Evaluar ubicacion de IMU centrada.
- Evaluar si MPU6050 alcanza o pasar a IMU mejor.

Firmware:
- Aprendizaje progresivo por datos.
- Deteccion de estados:
  - seco
  - agua
  - nadando
  - detenido
  - salida de agua
  - sin senal
  - posible riesgo
- Manejo de buffer cuando no hay senal.
- Envio resumido, no raw permanente.
- Reducir peso de telemetria para backend.

Apps:
- Laboratory: mantener como herramienta tecnica.
- Athlete: lectura entendible, resumen CSV y en vivo.
- Coach: monitor simple, mapa, multiple atleta, alerta fuerte.
- Family: simple, gratuita o casi gratuita, solo estado/ubicacion/alerta.

Graficos:
- Rotacion y simetria deben mostrar picos por lado.
- Alineacion corporal debe explicar simple.
- Impulso debe explicar que no es "fuerza absoluta", sino energia/cambio.
- Tendencia debe usar ventana de 30 segundos con barra de desplazamiento.
- Resumen debe poder entenderlo alguien sin saber sensores.
