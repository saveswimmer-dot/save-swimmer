Origen

La pregunta que abre el proyecto

Save Swimmer nace desde una necesidad simple: como saber si un nadador de aguas abiertas sigue activo, dentro de una zona segura y sin senales de riesgo cuando esta lejos de la orilla.

seguridadaguas abiertasfamilia / coach
Concepto

No es un reloj deportivo

Se define la vision: un wearable dorsal, ubicado entre escapulas, para interpretar contexto acuatico. El objetivo no es medir fitness, sino detectar compatibilidad o incompatibilidad con nado funcional.

dorsalIMUcontexto
Base tecnica

Primer firmware BLE + WiFi + Blynk

El ESP32-S3 comienza a leer el MPU6050, enviar JSON para Serial Studio, transmitir por BLE y reportar a Blynk. Sirve para demostrar lectura, telemetria y dashboard temprano.

ESP32-S3MPU6050BLEBlynk
Primer limite

Blynk y Serial Studio dejan de ser suficientes

El consumo de mensajes de Blynk y el limite de pruebas de Serial Studio empujan una decision importante: crear herramientas propias gratuitas para visualizar, grabar e interpretar datos.

costo mensualdatos propiosindependencia
App tecnica

Nace Field Viewer BLE

Se crea una app Android para conectarse al ESP por Bluetooth, ver telemetria viva, enviar perfil de atleta, iniciar sesiones y guardar CSV. La app pasa de demo a herramienta real de campo.

APKBluetoothCSV
Vision final

Se separan cuatro mundos de usuario

La app final se ordena en Nadador, Entrenador, Familia y Evento. El entrenador necesita mapa, geocerca y grupo activo; la familia necesita estados simples; el nadador revisa datos al salir.

nadadorcoachfamiliaevento
Producto

Se define la familia Lite / Events / Pro

LoRa sale de Lite porque requiere receptor propio. Lite se orienta a uso local con BLE, GPS/LTE futuro y membresia. Events puede usar infraestructura propia. Pro queda para satelital y travesias.

Save Swimmer |-- Lite: local, compacto, membresia accesible |-- Events: competencias, dashboard, posible base propia `-- Pro: travesias, satelital, mayor bateria
Hardware

Primer prototipo fisico

Se arma el prototipo con ESP32-S3, MPU6050, MT3608, TP4056, bateria, microSD y cableado. Todavia es grande, pero permite validar datos, energia, registro y ergonomia inicial.

prototipoenergiamicroSD
Datos

Los numeros empiezan a convertirse en lectura corporal

LR, FB, UD y MAG dejan de verse solo como ejes. Aparece la idea de rotacion, ritmo, impulso, loop de brazada, regularidad y comparacion contra la propia base del nadador.

brazadarotacionimpulsobaseline
MicroSD

La grabacion local se vuelve prioridad

La prueba muestra que el nadador se aleja del Bluetooth. Por eso el registro completo debe quedar en el dispositivo y recuperarse despues. Empiezan muchas versiones para estabilizar la microSD.

SD initreadbackflushsesion local
V036-V038

Se identifica el patron de fallo

Las sesiones cortan en momentos repetibles y el LED verde demora durante escrituras. Se documenta que el problema no es solo la app: hay bloqueo o inestabilidad en escritura/verificacion de la SD.

diagnosticofilas incompletasestado interno
V041-V044

La microSD llega a una version trazable

V041 logra una sesion larga pero con saltos. V042 y V043 mejoran integridad. V044 valida crecimiento real de archivo y marca pausas como #SD_GAP. Ya hay CSV util para analisis, aunque falta reducir pausas.

App demo

La vision comercial se separa de la app de pruebas

Se mantiene Field Viewer para laboratorio y campo, y se crea una app demo separada para mostrar la experiencia final. Esto permite probar datos reales sin mezclar la venta del concepto con diagnostico tecnico.

Campana

Save Swimmer entra a StartFund

Se arma el paquete de campana, portada, textos, recompensas, presupuesto y mensajes. StartFund aprueba el proyecto con meta de S/5,000 y envia imagenes promocionales para redes.

Presente

Lo que Save Swimmer es hoy

Un prototipo real con hardware funcional, app de campo, app demo, registro microSD, datasets, campana aprobada y feedback vivo. El siguiente foco es reducir pausas de SD, sensor de agua, GPS/LTE y pruebas reales en playa.
